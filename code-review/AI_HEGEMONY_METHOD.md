> 비전공 독학 개발자가 AI와 2시간 만에 앱을 만들면서 발견한,
> AI 협업의 주도권(헤게모니)을 유지하며 비용과 시간을 지키는 실전 전략.

- 48세, 사출금형 설계 23년 경력
- AI/프로그래밍 비전공, 완전 독학
- Claude API 잔액 $5.50로 프로젝트 시작
- 새벽 1시 반까지 개발 — 꽂히면 포기를 못하는 성격
- 돈 아까운 줄도 아는 사람

## 핵심 개념: AI 헤게모니

### "니가 만들지만, 헤게모니는 나에게 있다"

AI가 코드를 짜고, 버그를 고치고, "다 됐어"라고 말한다.
보통 사람은 여기서 바로 실행한다.

**나는 안 한다.**

AI가 "고쳤다"고 말했을 때, 그걸 곧이곧대로 믿고 실행하면
얼마나 많은 시간과 비용을 잃는지 알기 때문이다.

이것이 **AI 헤게모니** — AI를 도구로 쓰되, 최종 검증과 실행 결정은 반드시 내가 한다.

---

## 전략 1: "너는 나야" — 역할 전환 명령

### 실제 대화

> **나:** 이제부터 너는 나야. 내가 어떤 사람인지는 알지.
> 해뜨면 출근해야 되는데 새벽 1시반까지 이러고 있잖아.
> 꼽히면 포기를 못해. 근데 돈 아까운줄도 알아.
> 그래서 바로 빌링 페이지 가서 얼마 썼는지 보고 와서 쿠사리 주잖아.
> 근데 니가 고쳤다고 했는데 들어가서 실행해보고,
> **내 입장에서 답답해할 만한 걸 전수 조사해봐.**
> 너는 너지만 내 능력을 가진 너가 되는 거야.

### 왜 이게 강력한가

AI에게 "버그 찾아줘"라고 하면 기술적인 것만 찾는다.
AI에게 **"네가 나라면 뭐가 답답하겠어?"**라고 하면:

- 기술적 버그 (Windows 경로 문제 → FFmpeg 크래시)
- 비용 문제 (경고 없이 50센트 날리는 구조)
- UX 문제 (내보내기 성공인데 파일이 어딨는지 모름)
- 운영 문제 (자막 적용 두 번 누르면 중복)

**사용자의 맥락, 감정, 상황을 AI에게 주입하면 검증 품질이 완전히 달라진다.**

### 결과

AI가 스스로 자기 코드를 재검증해서 **12개 버그**를 추가 발견했다.
그 중 3개는 실행하면 **무조건 크래시**하는 치명적 버그였다.

## 전략 2: "실행하지 말고" — 검증 비용의 전환

### 실제 대화

> **나:** 대답부터 해줘. 그리고 인터벌 선택창 잘 안 보여. 가독성 제로야 ㅋㅋ
>
> **나:** (빌링 스크린샷) 일단 39센트 수업료 냈고
>
> **나:** 너한테 검증 시키면 토큰 안 드는 거잖아. 실행해서 터지면 API 토큰 날리는 거고.

### 핵심 원리

| 방식 | 비용 | 시간 |
|------|------|------|
| 바로 실행 → 에러 → 수정 → 재실행 | API 토큰 ($$) | 길다 |
| AI에게 "너가 나라면?" → 코드 리뷰 | 0원 | 짧다 |

**검증 비용을 달러(API 호출)에서 AI 연산(무료)으로 전환시키는 것.**

이건 금형 설계에서 **시사출 전 도면 검토**와 같은 원리다.
금형 깎고 시사출 해봤더니 언더컷 있으면? 수정 비용이 도면 단계에서 잡는 것과 비교가 안 된다.

---

## 전략 3: "시드머니를 지켜라" — 제한된 자원의 전략적 운용

### 상황
- API 크레딧: $5.50 (전부)
- Vision API 1회 호출: ~$0.02~0.03
- 프론트엔드 버그로 결과 표시 실패 → 토큰만 날림: 39센트

### 대응
1. 빌링 페이지 스크린샷으로 현재 잔액 확인 ($4.61)
2. AI에게 비용 경고 기능 요구 → `confirm()` 팝업으로 예상 비용 표시
3. "수업료 냈다"는 표현 — 실패를 학습으로 전환

### 원칙

> "헤게모니로 시드머니를 지키는 전략"

돈이 무한하면 막 써도 된다. 하지만 $5.50이 전부일 때,
**매 실행 전에 "이게 돈값을 하는 실행인가?"를 판단하는 것은 사람의 일**이다.
AI는 돈 개념이 없다. 시키면 시키는 대로 API를 호출한다.

## 전략 4: "너는 내가 될 수 있지만, 나는 니가 될 수 없다" — 역할 분담의 철학

### 실제 대화

> **나:** 너는 내가 될 수 있지만 나는 니가 될 수는 없잖아.
> 이 정도 했으면 믿어야지 ㅋㅋ
> 이다음에 문제 해결이 진짜 티키타카가 되는 거 아니겠어?

### 해석

- **AI는 사용자가 될 수 있다** — 역할 주입으로 사용자 관점에서 검증 가능
- **사용자는 AI가 될 수 없다** — 코드 생성, 대량 분석은 AI만 가능
- **그래서 역할을 나눈다:**
  - 사용자: 방향 설정, 품질 검증, 실행 결정
  - AI: 코드 생성, 자기 검증, 버그 수정

### 티키타카의 조건

AI에게 한번 "너는 나야"를 시키고 나면,
AI는 사용자의 기준(돈, 시간, UX)을 학습한 상태가 된다.
그 이후부터는 사용자가 말 안 해도 AI가 먼저 체크한다:

- "이거 Windows에서 경로 터지나?"
- "이거 토큰 얼마 드나?"
- "이거 끝나고 사용자가 파일 찾을 수 있나?"

**한 번의 역할 전환이 이후 모든 작업의 품질을 올린다.**

---

## 전략 5: "순서를 역행해도 되는 판단" — 언제 AI를 믿을지의 기준

### 맥락

처음에는 "기존 편집기 쓰는 게 낫지 않냐"고 물었다.
AI가 "클로드 비전으로 화면만 보고 자막 생성이 가능하다"고 했을 때:

> **나:** 이러면 얘기가 달라지는데. 순서를 역행해도 해볼 만한 거 같은데.

### 판단 기준

1. **기존에 없는 것인가?** → 음성 없는 화면 캡처에서 자막 자동 생성 = 없음
2. **내 상황에 맞는가?** → 크몽 포트폴리오 시연 영상 = 음성 없는 화면 캡처
3. **위험 감수 가능한가?** → $5.50 시드, 실패해도 학습 가치 있음

순서를 역행하는 것(편집기 사용법 익히기 전에 직접 만들기)은
**명확한 차별점이 있을 때만** 정당화된다.

## 방법론 요약

### AI를 잘 쓰는 사람의 5가지 원칙

1. **헤게모니를 유지하라** — AI가 "됐다"고 해도 내가 검증한다
2. **역할을 전환하라** — "너는 나야"로 AI에게 내 관점을 주입한다
3. **검증 비용을 전환하라** — 실행(돈)이 아닌 코드 리뷰(무료)로 검증한다
4. **시드머니를 지켜라** — 제한된 자원일수록 매 실행의 가치를 판단한다
5. **역할을 분담하라** — 방향은 내가, 실행은 AI가, 검증은 같이

### 금형 설계자의 비유

> 도면 검토 없이 시사출 때리는 건 바보짓이다.
> AI가 짠 코드를 검증 없이 실행하는 것도 똑같다.
> **시사출 전 도면 검토 = 실행 전 AI 자기 검증.**
> 이 한 단계가 시간과 돈을 지킨다.
