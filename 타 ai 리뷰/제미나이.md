요청하신 설정에 따라 철저히 외부 배경을 배제하고, 오직 제공된 코드와 아키텍처만을 바탕으로 냉정하고 날카롭게 프로젝트(CutSense)를 분석했습니다.

현재 이 코드는 1인 개발 프로토타입으로는 작동할 수 있으나, **실제 환경(특히 대용량 영상 처리 및 API 통신)에서는 서버가 멈추거나 예상치 못한 오류를 뱉어낼 치명적인 결함**들을 안고 있습니다. 

다음은 시스템 전반에 걸친 문제점과 개선 사항입니다.

### 1. 치명적 버그 및 아키텍처 결함 (Critical Issues)

**1) FastAPI 비동기 이벤트 루프 차단 (가장 심각함)**
* **문제:** `main.py`의 엔드포인트들(`/api/cut`, `/api/subtitle`, `/api/export`)은 `async def`로 선언되어 있습니다. 그러나 내부에서 호출하는 `video_processor.py`의 FFmpeg 실행 로직(`subprocess.run`)은 **완벽한 동기(Synchronous) 블로킹 코드**입니다.
* **결과:** FFmpeg가 영상을 인코딩하는 5~10분 동안 FastAPI의 이벤트 루프가 완전히 멈춥니다. 이로 인해 기껏 구현해 둔 WebSocket(`/ws/progress`)은 제대로 된 실시간 진행률을 브로드캐스트하지 못하며, 프론트엔드는 응답 지연으로 타임아웃 오류를 일으킬 확률이 99%입니다.
* **해결:** `subprocess.run` 대신 `asyncio.create_subprocess_exec`를 사용하거나, 기존 코드를 유지하려면 `fastapi.concurrency.run_in_threadpool`로 감싸서 별도의 스레드에서 실행해야 합니다.

**2) Windows 환경의 FFmpeg Subtitle 필터 경로 오류**
* **문제:** `video_processor.py`의 `burn_subtitles` 함수에서 Windows 경로 문제를 피하고자 `srt_escaped = srt_path.replace("\\", "/").replace(":", "\\:")` 처리를 했습니다. 하지만 `subprocess.run`에서 `-vf f"subtitles='{srt_escaped}'"` 처럼 리스트 요소 내부에 작은따옴표(`'`)를 직접 포함시켰습니다.
* **결과:** Windows의 `subprocess.run` (shell=False 기본값) 환경에서는 이 작은따옴표가 쉘에 의해 해석되지 않고 파일 경로의 일부로 그대로 FFmpeg에 전달되어 **"No such file or directory"** 오류를 발생시킵니다.
* **해결:** 리스트 형태로 인자를 넘길 때는 필터 문자열 내부의 작은따옴표를 제거하거나, 이스케이프 방식을 완전히 재설계해야 합니다.

**3) 치명적인 보안 취약점 (Path Traversal)**
* **문제:** `main.py`의 다운로드 엔드포인트에서 파일명 검증이 전혀 없습니다.
    ```python
    @app.get("/api/download/{filename}")
    async def download_file(filename: str):
        file_path = os.path.join(UPLOADS_DIR, filename)
    ```
* **결과:** 사용자가(혹은 공격자가) `filename`에 `../../../Windows/System32/cmd.exe` 같은 값을 넣으면 서버 시스템의 파일을 그대로 탈취할 수 있습니다. 
* **해결:** `filename = os.path.basename(filename)` 처리로 디렉토리 이동 공격을 차단해야 합니다.

### 2. API 비용 효율성 및 Vision AI 로직

**1) 토큰 및 요금 폭탄 가능성**
* **문제:** `ai_service.py`에서 2초 간격(`SMART_INTERVAL = 2.0`)으로 프레임을 추출합니다. 4분짜리 영상이면 약 120장의 이미지가 추출됩니다. 이를 6장씩 배치로 묶어 Pass 2에서 Claude/Grok으로 전송합니다.
* **결과:** Claude 3.5 Sonnet은 이미지 해상도(현재 1280px로 리사이징)에 따라 토큰을 산정합니다. 120장의 이미지는 수십만 개의 Vision 토큰을 소모합니다. 4분 영상 기준 $0.50 이내로 맞추기에는 매우 아슬아슬하거나 초과할 위험이 높습니다.
* **해결:** 무조건 2초 간격으로 캡처하는 방식은 비효율적입니다. FFmpeg의 `select='gt(scene,0.3)'` 필터를 사용하여 **실제 화면(장면)이 전환되는 시점의 프레임만 추출**하도록 변경하면 API 호출 횟수와 비용을 획기적으로(80% 이상) 줄일 수 있습니다.

**2) Base64 인코딩의 네트워크 오버헤드**
* 모든 프레임을 Base64 문자열로 변환하여 메모리에 올린 뒤 JSON으로 전송하고 있습니다. 배치 1개당(6프레임) 페이로드 크기가 5~10MB에 달합니다. 이는 메모리 스파이크를 유발하고 API 통신 지연을 초래합니다.

### 3. 프론트엔드 (React + Babel) 한계

**1) 브라우저 런타임 컴파일의 성능 저하**
* **문제:** `index.html`에서 `<script type="text/babel">`을 사용해 1,000줄이 넘는 React 코드를 브라우저가 접속할 때마다 실시간으로 트랜스파일링하고 있습니다.
* **결과:** 초기 로딩 속도가 심각하게 느려지며, 코드가 길어질수록 메모리 누수와 브라우저 버벅임 현상이 심해집니다.
* **해결:** 프로토타입 단계가 끝났다면, Vite나 Create React App(CRA)를 사용해 빌드된 정적(static) JS 파일을 제공해야 합니다. 데스크톱 앱(Electron)으로 전환하기 위해서도 Node.js 기반의 빌드 시스템 도입이 필수적입니다.

**2) 취약한 상태 관리**
* 새로고침 시 자동 복원 로직이 `useEffect` 단 하나에 의존하고 있으며, `uploadedFilePath`와 `videoFile`의 URL 객체(blob) 매핑이 불안정합니다. 새로고침 후 `URL.createObjectURL()`이 재생성되지 않으면 화면이 까맣게 나오거나 재생되지 않는 현상이 발생할 수 있습니다.

### 4. FFmpeg 처리 안정성 (대용량 영상)

**1) 세로 영상(9:16) 변환 필터 그래프의 무거움**
* `export_video`의 세로 변환 필터(`boxblur=20:5`)는 CPU 연산을 극도로 많이 요구합니다. 10분 이상의 대용량 영상 처리 시 600초 타임아웃(`timeout=600`)에 걸려 내보내기가 강제 종료될 확률이 높습니다. 하드웨어 가속(NVENC 등)을 명시하지 않아 오직 CPU 코어만 혹사시키는 구조입니다.
